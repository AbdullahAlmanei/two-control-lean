# Daily log

## Goal
Open a new paper-only formalization track for the Clifford Lemma 11 result inside the existing Lean repo, identify only the dependencies actually needed for the two-qubit theorem, generate the dependency and blueprint artifacts, and refactor the Lean statement layer so the later proof work can reuse the repo's existing matrix and unitary helper ecosystem cleanly.

## What I changed
The first step was scoping. I read the shared migration context, then compared the Clifford paper against the current Lean project to determine which earlier facts were already present and which ones would need new local statements. The key reuse point is the existing two-qubit cosine-sine theorem `TwoControl.CosineSine.cosinesine_exists`, which serves as the imported specialization of paper Lemma 2. I also clarified the proof boundary for this track: although paper Lemma 1 matters to the full `n`-qubit story, it is not explicitly used in the proof of the two-qubit Lemma 11, so this first track intentionally excludes the recursive argument.

After that scoping pass, I created the Clifford-specific migration artifacts under `docs/migration/clifford/`. These include the paper packet, theorem map, generation packet, and a dependency graph that isolates the exact proof cone for Lemma 11. The resulting dependency chain is: imported two-qubit Lemma 2, paper Lemma 3 (`R_y` via `H`, `S`, and `R_z`), the two-qubit specialization of paper Lemma 4, a controlled-diagonal or controlled-`R_z` support proposition, a one-qubit exact synthesis proposition, and finally Lemma 11 itself. The intent of these artifacts is to lock the proof scope before pushing more Lean structure.

I then added a first blueprint chapter for this track at `blueprint/src/chapters/clifford_lemma11.tex`, registered its Lean declarations in `blueprint/lean_decls`, and included it from `blueprint/src/content.tex`. The blueprint chapter states the imported cosine-sine node, the new Clifford nodes, and the support propositions in a paper-facing order. This gives a readable dependency story before any proof development begins.

On the Lean side, I added a new `TwoControl.Clifford` namespace with `Definitions`, `Statements`, and `Main` modules, then imported that surface from the repo roots. The initial version introduced the relevant one- and two-qubit gate matrices, a small primitive gate language for the target synthesis statements, and theorem stubs for the Clifford-paper nodes. Once that basic track compiled, I refactored the statement surface to be more proof-ready and closer to the helpers already present elsewhere in the repo.

That refactor is the main technical improvement of the session. Instead of keeping only coarse existential circuit theorems, I introduced interfaces that match the actual proof ingredients and the current helper library better. In `TwoControl/Clifford/Statements.lean`, the statement layer now includes `one_qubit_diagonalization`, which directly reuses the existing `unitary_diag2_exists` theorem, and `controlled_diag_via_phase_and_controlled_rz`, which exposes the exact controlled-diagonal factorization needed for the `lemmadiag` step in a form compatible with `controlledGate`, `diag2`, and local tensor-factor gates. In `TwoControl/Clifford/Definitions.lean`, I added `phaseShift`, `reverseControlledGate`, and circuit-composition lemmas for the one- and two-qubit primitive lists, including append and lift lemmas. These are there specifically so later proofs can assemble local exact syntheses into the full Lemma 11 circuit without repeatedly unfolding the list semantics by hand.

The current state is a clean statement-layer scaffold for the Clifford Lemma 11 track. The paper-facing theorem names remain explicit, the support propositions are in place, and the helper layer is aligned with the repo's existing matrix decomposition and tensor manipulation infrastructure. The proofs themselves are still intentionally deferred with `sorry`; the work so far is about getting the dependency story and interfaces into the right shape before starting the actual proof scripts.

## What worked
The strongest decision was to treat this as a separate paper-only migration track rather than trying to force it into the original two-control proof flow. That avoided false dependencies on unrelated Rocq material and kept the scope narrow. Reusing `TwoControl.CosineSine.cosinesine_exists` as the imported two-qubit Lemma 2 was also the right anchor: it gave the whole track a solid, already-proved starting point inside the repo.

The later statement refactor also worked well. The original stub layer was too coarse because it jumped straight to final circuit-existence statements, which would have made the eventual proofs fight the helper library. Replacing that with matrix-facing intermediate theorems and simple circuit-composition lemmas produced a much better fit with the existing diagonalization, controlled-gate, and Kronecker-product lemmas already available in the project.

## What failed
The first pass at the Clifford Lean stubs was not yet proof-oriented enough. In particular, statements that only asserted final circuit existence did not expose the right intermediate structure for later proofs, so they had to be redesigned. There were also a few build-level issues during the refactor, including import-path mismatches for complex exponentials, name-qualification mistakes, and proof-script issues in the new append and lift lemmas for circuit semantics. Those were local problems rather than conceptual blockers, but they confirmed that the build, not the editor, is the real validation step for this repo.

## Reusable lesson
For a new paper track inside an existing formalization, the critical move is to separate paper-facing theorem statements from proof-facing helper interfaces early. If the statement layer only records the final theorem shape, the later proof work often becomes harder because the repo's reusable lemmas live at a more structural matrix level. It is better to keep the paper nodes visible while also adding exact helper statements that match the existing abstractions the proofs will actually use.
